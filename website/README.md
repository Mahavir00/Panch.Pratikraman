# Pañca Pratikramaṇa — Website

A reverent, three-language (**English · ગુજરાતી · हिन्दी**) website presenting the **Pancha
Pratikramana** sūtras of the **Achhalgach (Anchalgachchha / Vidhipakṣa)** Śvetāmbara Mūrtipūjaka
Jain tradition, with verse-by-verse commentary.

It is a **client-side-only** React + TypeScript + Vite app. All content is read from static JSON
generated by the translation tool in the repository root (`../data`). Nothing is hard-coded — the
**scripture is the single source of truth**.

> Live build target: `https://<user>.github.io/Panch.Pratikraman/`

---

## Quick start

```bash
cd website
npm install
npm run sync-data     # build the public data bundle from ../data  (re-run anytime data changes)
npm run dev           # start the dev server
```

Open the printed local URL. The app uses **HashRouter**, so deep links look like
`…/#/s/nandisutrani-pratham-sajay`.

---

## The data pipeline is ongoing — re-sync often

The translation tool keeps generating and updating JSON under `../data/translations/…`. Whenever it
does, refresh the website's data bundle:

```bash
npm run sync-data         # one-shot, idempotent, tolerant of partial/missing data
npm run sync-data:watch   # re-sync automatically on any change under ../data
```

`sync-data` reads `../data` and writes a clean, versioned bundle to **`website/public/data/`**:

| File | Contents |
|---|---|
| `index.json` | structure tree (pratikramans + sequences + overrides), sūtra registry, āvaśyakas, appendix, tradition aliases, and a derived **availability map** (per-sūtra translation progress per language) |
| `sutra/<id>.json` | one lazy-loaded bundle per sūtra: canonical verses × up-to-three prefaces × all per-shloka translations, merged by `shlokaId × language` |
| `glossary.json` | the three language glossaries + `pos-labels` |
| `search-index.json` | compact client-side search index |
| `achhalgach.md` | the tradition knowledge base (rendered on the About page) |
| `pdfs/*` | any complementary PDFs found in `../data/pdfs` |

It prints a summary of counts and what is still missing. **The bundle is git-ignored** (regenerated),
so always run `sync-data` before `dev`, `build`, or `deploy`.

## Validate

```bash
npm run validate          # asserts the synced bundle is internally consistent (run after sync-data)
```

Checks: structure referential integrity (every referenced `sutraId` exists; empty-sequence
pratikramans declare `basedOn` + `overrides`); every `sutra/<id>.json` parses and non-pending
shlokas carry `native_script`; a script spot-check (Gujarati/Hindi prose ≥ 85% target script,
English recitation in Latin/IAST); and that no route target is missing from the bundle.

## Build & preview

```bash
npm run build             # tsc + vite build  → website/dist
npm run preview           # serve the production build locally under the GH-Pages base
```

The Vite **`base`** is set to `'/Panch.Pratikraman/'` (it must match the repository name). Every
asset and data fetch goes through `import.meta.env.BASE_URL` — never an absolute `/…` path — so the
site works when served from a project sub-path.

## Deploy to GitHub Pages

Two supported paths — use either.

### A. GitHub Actions (recommended, automatic)

`.github/workflows/deploy-pages.yml` (in the repo root) runs on every push to `main`. It installs,
`sync-data`s, validates, builds, and publishes to Pages. **Enable it once:** repository
**Settings → Pages → Build and deployment → Source → GitHub Actions**.

### B. One command from your machine

```bash
cd website
npm run deploy            # runs predeploy (sync-data + build) then publishes ./dist with gh-pages
```

This pushes the built site to the `gh-pages` branch. Set **Settings → Pages → Source → Deploy from a
branch → `gh-pages` / root** if you use this path.

> **Why HashRouter + `base`?** GitHub Pages serves static files with no SPA fallback. HashRouter keeps
> all routing in the URL fragment, so deep links resolve with zero server config; the `base` ensures
> assets load from the `/Panch.Pratikraman/` sub-path.

## Project layout

```
  scripts/
    sync-data.mjs       # build public/data from ../data   (npm run sync-data)
    validate-data.mjs   # validate the bundle              (npm run validate)
  public/

    state/              # language / theme / library (bookmarks) contexts + useAsync
    config.ts           # BASE_URL helpers, language metadata
    data.ts             # cached fetchers for the bundle
    types.ts            # data-model types (lockstep with sync-data.mjs)
    index.css           # design tokens (light/dark), Jain palette, component classes
```

## Features

- **Three scripts**, self-hosted (Noto Serif/Sans Gujarati & Devanagari + EB Garamond + Inter).
- **Global language toggle** (EN / ગુ / हि), persisted to `localStorage`.
- **Progressive disclosure** per verse: original → recitation → plain meaning → idiomatic/literal →
  word-by-word → commentary/doctrine/practice/cross-refs/sources.
- **Glossary tooltips** inline across commentary and prose; a searchable Glossary page.
- **Search** (⌘/Ctrl-K) across sūtras, verses, and glossary; **bookmarks** & **recently viewed**.
- **Deep-linkable, shareable** verse URLs with copy-link buttons.
- **Light / dark** themes; **print** stylesheet; a **`/style`** design-system showcase.
- **Partial-data tolerant**: missing translations render a tasteful "in progress" state, never an
  error; per-language completion meters everywhere.

## Notes

- Do not edit files under `../data/book`, `../data/corpus/pratikraman-structure.json`, `../index.js`,
  `../src`, or `../prompts` — those belong to the generation tool. This app only **reads** the data.
- `public/data` is git-ignored by design (it is regenerated). If you prefer to commit a snapshot for
  reproducible deploys, remove the `public/data` line from `website/.gitignore`.
